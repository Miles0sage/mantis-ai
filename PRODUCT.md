# Mantis — Product Brief

## Positioning

Mantis is a coding product for bounded engineering work.

This is not an "agent OS" pitch.
This is "read the repo, make the change, prove it, and show the work."

## Wedge

**Tagline:** The coding agent for bounded engineering work.

## Why It Matters

Most coding agents optimize for raw model quality or terminal parity.
Very few optimize for:

- low-cost execution by default
- bounded task reliability
- browser-based oversight
- background task execution
- hard budget ceilings
- checkpoint and resume
- isolated workers
- visible plan, diffs, and approvals
- verifier-backed completion

That is the gap Mantis should own.

## Mantis V1

The minimum product that feels real:

1. Background jobs you can queue and revisit
2. Visual task tree and live streaming
3. Exact spend meter with hard stop budgets
4. Checkpoint and resume
5. Approval flow for destructive actions
6. Multi-file edit preview before apply
7. Cheap planner plus strong writer routing

## Model Policy

- Cheap model handles planning, file search, context gathering, and simple edits
- Strong model handles cross-file synthesis, ambiguous debugging, and high-risk edits
- Escalation is explicit, not default

## Product Standard

Mantis should be able to:

1. take one concrete coding task
2. choose the cheapest safe execution path
3. isolate risky multi-file work
4. verify the result
5. expose the diff, worker ownership, and decision trail

## What We Should Not Claim

- broad autonomous software company behavior
- universal reliability on arbitrary repo surgery
- solved model latency for open-ended tasks

The honest position is narrower and stronger: Mantis is a practical coding operator for tasks that can be scoped, checked, and reviewed.

## What To Avoid

Do not build these before the core product works:

- giant multi-agent swarms
- complex vendor-specific transport layers
- novelty UI features
- expensive always-on premium model defaults
- architecture that only improves demos, not daily use
